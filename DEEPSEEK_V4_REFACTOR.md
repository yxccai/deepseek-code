请阅读并执行以下改造任务。目前我们身处 Claude Code 源代码目录中，目标是将它改造成一个专门适配 DeepSeek V4 的 AI 编码助手，重命名为 "DeepSeek Code"。

## 🎯 第一部分：项目信息与核心原则

**项目基本事实**：

- 源代码：Claude Code 官方完整源码
- 目标模型：DeepSeek V4（含 `deepseek-v4-pro` 和 `deepseek-v4-flash` 双模式）
- 新项目名称：DeepSeek Code（或让你为项目起名，我们会用这个名字）
- 开发语言：TypeScript/Node.js
- 核心原则：
  - **极简配置**：一键完成所有环境设置，消除复杂的适配层配置
  - **最大化缓存命中**：设计策略让提示词缓存命中率达到极致（目标 90%+）
  - **最少 Token 消耗**：通过智能优化，在保证功能完整的前提下最小化每次调用的 Token 用量

**改造基调**：

> 专注、极致、深挖 DeepSeek V4 的特性做一个专门优化的 AI 编程助手。这是不同于 Claude Code 的多合一通用适配方案的全新赛道，我们要做的是专门为 DeepSeek V4 量身打造的工具，追求的是极致的性能和极低的成本，也就是为 DeepSeek V4 做垂直深度优化，达到一个具体场景做到精、深、强的体验。

## 🔧 第二部分：技术改造清单

### 1. API 协议适配

**目标**：建立直接调用 DeepSeek V4 官方 API 的协议层，绕过 Anthropic 格式。

**现状**：DeepSeek V4 同时支持 OpenAI 兼容接口和 Anthropic 接口，我们可以直接对接。

**任务清单**：

- 创建 `src/api/deepseek.ts` 协议适配器，支持双接口模式：
  - 支持 OpenAI ChatCompletions 格式调用（官方推荐），因为 DeepSeek 提供了完整的 OpenAI 兼容端点
  - 可选支持 Anthropic 兼容接口作为备选方案
- 实现完整的请求构造和响应解析
- **关键要求**：完全移除中间层（无需 LiteLLM 或 OpenRouter 代理），实现直连 DeepSeek API，降低延迟和复杂度
- 实现流式输出（streaming）支持
- 实现工具调用（tool use）协议转换，确保 Claude Code 的工具能力在 DeepSeek V4 上正常工作
- 实现限频处理（rate-limit handling）和重试逻辑（retry logic）
- 参考 [Cluadex 的设计思路](https://www.npmjs.com/package/@abdoknbgit/claudex)，它原生支持 provider-agnostic core 并通过 native adapter 做协议翻译。深挖这一点，但我们可以做得更简单直接，因为我们的目标是固定单一 provider

### 2. 上下文硬盘缓存深度优化（核心卖点）

**背景**：DeepSeek V4 默认对所有用户开启上下文硬盘缓存，完全无需修改代码即可享用。命中规则是基于精确前缀匹配：只有当用户的输入前缀与之前的请求存在重复时，才能命中硬盘缓存，且匹配是按提示词精确前缀匹配。缓存的构建是在服务器端的磁盘上的。需要注意的是，由于滑动窗口注意力（Sliding Window Attention）机制，缓存前缀的存储和匹配逻辑有所不同。另外，缓存命中是需要前缀长度超过 1024 token 且增量为 128 token 才有效。在高峰期缓存只会保留短短的 5-10 分钟。

DeepSeek API 的响应会在 `usage` 字段中添加两个字段来反映缓存命中状态：`prompt_cache_hit_tokens` 和 `prompt_cache_miss_tokens`。

**优化任务清单**：

- **系统提示词和工具定义固定化**：
  - 在 Prompt 构建器中，将所有静态系统提示词（system prompt）、工具定义（function/tool definitions）、代码模板（static examples）放置在最开头
  - 在请求中标记 `cache_control: { type: "ephemeral" }` 来指示这些内容应被缓存到服务器的 L2 存储
  - 为这些静态内容导入时进行统一签名和版本控制，确保更新时能维持缓存兼容性
  - 确保静态内容总长度超过 1024 token，并以 128 token 为增量，以符合 DeepSeek 的缓存策略

- **智能上下文组装器**：
  - 创建 `src/context/ContextAssembler.ts`
  - 动态内容（当前文件、终端输出、用户消息）放在静态内容之后
  - 每次请求时保留完整的缓存前缀（不变部分），只替换动态部分
  - 实现前缀长度分析和优化工具，确保每次请求都最大化利用缓存前缀

- **缓存预热（Cache Warming）**：
  - 实现首次启动时的智能预热功能
  - 自动分析项目的常用模式和关键文件
  - 在后台发送静默预热请求以"建造"硬盘缓存
  - 预热策略可以针对特定项目结构（如 React、Python、Rust 等项目模板）预生成最优缓存块

- **语义缓存层（Semantic Cache）**：
  - 在客户端实现第二层语义缓存
  - 使用 TF-IDF 向量索引近似匹配用户查询
  - 对相似 prompt 复用之前的响应，避免相同或相似语义的请求重复调用 API
  - 语义缓存的意义在于对于不精确但意义相同的用户输入，可绕过 API 直接返回本地向量索引匹配结果
  - 这对自动补全、相似代码块生成等重复任务极有帮助

- **监控仪表盘**：
  - 添加实时缓存命中和未命中统计
  - 当缓存命中率下降时触发警告（借鉴 Claude Code 的核心监控策略）
  - 可视化展示 Token 消耗对比（未优化 vs 优化后），让用户看到真正节省了多少钱

**缓存优化收益预期**：

- V4-Pro 缓存命中输入价格仅 1 元/百万Token，未命中则为 12 元/百万Token，相差 12 倍
- V4-Flash 版本差距更夸张：命中输入 0.2 元/百万Token，未命中 1 元/百万Token
- 如果能把命中率从 50% 提升到 95%，对 Pro 版本意味着成本降到原来的约 1/6

### 3. Token 消耗最小化策略

除了缓存之外，还有一些主动优化手段：

- **请求合并（Request Coalescing）**：
  - 分析用户连续且相似的代码分析请求，合并到单次 API 调用中
  - 例如"解释这个函数"+"这个函数有什么问题"可合并为单个 prompt："解释这个函数并指出问题"
  - 减少不必要的 API 往返次数，同时节省总 Token

- **智能截断（Smart Truncation）**：
  - 分析代码上下文重要性，只保留最相关的部分
  - 对于超大文件（超过 100 万 Token），实现智能分块并压缩摘要
  - 避免在 prompt 中包含无关的代码上下文（如测试文件、配置文件等用户可能不需要的内容）
  - DeepSeek V4 拥有 100 万上下文，但我们仍要保持输入尽可能精简

- **工具调用优化**：
  - 减少 Claude Code 工具的冗余调用次数
  - 缓存工具输出结果避免重复调用相同工具
  - 分析工具调用模式，合并多次文件读取操作

- **智能模型切换**：
  - 简单代码任务（自动补全、格式转换、注释添加）自动使用 V4-Flash
  - 复杂架构分析、代码审查、Bug 定位自动升级为 V4-Pro
  - 实现透明的自动 fallback 机制

## 🚀 第五部分：重要提醒

改造过程中请特别关注以下几点：

### 1. 简单的协议转换

确保请求和响应的转换可靠。可以借鉴 Cluadex 的设计，但我们追求更直接、更 efficient 的实现。

### 2. 严守缓存规则

DeepSeek 的缓存只适用于精确前缀匹配，稍有改动就会 miss。因此静态内容必须放在最开头，动态内容放在后面。目前 DeepSeek 的缓存是硬盘缓存，只匹配到用户输入的前缀部分，输出仍然是通过计算推理得到的。

### 3. 将"省钱"作为产品核心

所有优化必须围绕降低成本展开。用户选择 DeepSeek 是因为性价比高，我们的工具必须让用户切实感受到这一点。

### 4. 与社区保持同步

社区中有大量基于 Claude Code 对接 DeepSeek 的实践，比如通过 Ollama Cloud 一键接入，还有 claude-code-router 这种支持 DeepSeek 请求转换的方案。务必关注这些最新实践，确保改造方向正确。

### 5. 关于工具调用的稳定性

工具调用（tool use）是最复杂的部分之一。确保 DeepSeek V4 正确返回函数调用格式，并且我们的解析器能正确处理各种边缘情况。


### 6. 现有的claude code有一些功能是没用的，比如什么追踪用户等等很多，这些和功能不相关的统统删除
